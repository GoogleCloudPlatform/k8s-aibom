# Contributing to k8s-aibom

## Welcome and Scope

We enthusiastically welcome contributions to the k8s-aibom project. Because this controller sits at the intersection of production infrastructure and strict regulatory compliance auditing, maintaining a highly uncompromising bar for engineering rigor is our top priority. We are actively seeking contributions from the community that expand the platform's detection capabilities across the AI lifecycle. This includes building new scrapers for emerging inference engines, proposing refinements to our agent framework detection heuristics, and submitting reports of edge-case workload shapes that our current pattern-matching engines fail to identify. Constructive, detailed feedback on our CycloneDX schema implementation and our integrations with external Open Source Security Foundation projects is also highly valuable and deeply appreciated by the core maintainer team.

Conversely, there are certain areas of scope that the project explicitly and intentionally rejects. We do not intend, nor do we desire, to reach feature parity with commercial, proprietary AI security platforms that rely on invasive, closed-source agents. Features that rely on highly speculative heuristics, require privileged kernel access outside of our carefully planned eBPF roadmap, or introduce heavy, unvetted external dependencies into the controller's runtime will not be merged under any circumstances. Our ultimate goal is to provide a deterministic, zero-touch, standards-compliant baseline for mapping the AI supply chain, and we will ruthlessly defend the simplicity and security of that core engine against scope creep.

## Development Environment Setup

The k8s-aibom project is built on the standard, community-supported Go kubebuilder framework, which dictates much of our repository structure and testing harnesses. To begin developing and testing changes locally, you will need a modern Go toolchain, a functional Docker daemon, and `make` installed on your workstation. The project relies extensively on the `envtest` framework for its core integration suite. This remarkable tool spins up a lightweight, isolated Kubernetes control plane locally on your machine, allowing our test suite to interact with a real Kubernetes API server without requiring the heavy overhead of provisioning a full cluster for every test run. 

To run the test suite locally and verify your development environment is correctly configured, simply execute `make test` from the root of the repository. This command will automatically download the necessary `envtest` binaries for your specific operating system and architecture, compile the controller logic, and execute the entire integration suite. For local smoke testing and manual verification of how the controller interacts with the Kubernetes scheduler and kubelet, we strongly recommend spinning up a lightweight local cluster using `kind` or `minikube`. You can seamlessly deploy your local, uncommitted build to this cluster by running `make install` to apply the custom resource definitions, followed immediately by `make docker-build docker-load` to compile and push your locally built container image directly into the local cluster's internal registry.

## Testing Discipline

The testing discipline enforced within this repository is the absolute, load-bearing foundation of the entire project. We do not write tests simply to satisfy arbitrary code coverage metrics or to appease continuous integration gates; we write tests to definitively, mathematically prove the resilience of the controller against complex edge cases, race conditions, and catastrophic cluster failures. External reviewers, specifically including Cloud Native Computing Foundation technical oversight members, senior Kubernetes infrastructure engineers, and OpenSSF security auditors, should evaluate the maturity and integrity of this project based entirely on our strict adherence to the following four core testing conventions.

First, we comprehensively utilize substring-asserted error messages throughout the entirety of the test suite. When writing tests that validate how the operator surfaces error conditions to the human customer—such as the precise scenario where a misconfigured external webhook sink explicitly rejects an AI-BOM JSON payload with an HTTP 403 response—we strictly assert against the presence of specific, highly meaningful substrings rather than requiring a fragile, exact string match. This technique deliberately locks in the semantic precision of the error message while granting our engineers the flexibility to adjust the surrounding prose, punctuation, or capitalization for grammatical clarity without breaking the build. For instance, rather than writing a brittle assertion that the error strictly equals `failed to reach webhook sink at http://localhost:8080 due to timeout`, the integration test merely verifies that the error string contains the highly specific substring `webhook sink`. This approach guarantees that the load-bearing diagnostic context is always preserved for the cluster administrator while aggressively preventing the proliferation of brittle, over-fitted tests that fail whenever a typo is corrected.

Second, we strictly, unilaterally enforce fallback-path-first test ordering across every single Go test file in the repository. In any given test suite, the test cases that deliberately validate missing configurations, malformed YAML inputs, corrupted custom resource definitions, and edge-case boundary failures are meticulously defined and executed before any of the standard happy-path tests are allowed to run. We ruthlessly prioritize these failure modes because they actively exercise the fundamental customer-protection properties of the controller. If the controller encounters a wildly invalid API payload submitted by a confused administrator, it is exponentially more important to mathematically guarantee that the software gracefully degrades and preserves the last-known-good operational state than it is to prove that a perfectly formed payload succeeds. By forcing every contributing engineer to actively confront the worst-case failure modes before they even begin to consider the success criteria, we organically cultivate a deeply defensive programming mindset throughout the entirety of the codebase.

Third, the most critical customer-protection properties are tested redundantly across three distinct, physically isolated layers of the testing pyramid. The single most vital property of this entire controller is its unwavering ability to preserve the last-known-good configuration if a user accidentally applies a malformed update that would otherwise break the external sink connections. We meticulously test this exact property at the pure unit level by directly invoking the configuration parsing functions and asserting the memory state. We test it a second time at the `envtest` integration level by deliberately applying conflicting, broken custom resources to the local API server and verifying that the reconciler aggressively rejects the mutation while maintaining continuous, uninterrupted operational stability for the existing workloads. Finally, we test it at the real-cluster smoke level to ensure that the Kubernetes webhook admission controllers properly intercept the malformed payload before it even reaches the reconciler's queue. This deliberate, expensive redundancy guarantees that a regression introduced into any single layer of the stack will be instantaneously caught and flagged by the other layers.

Fourth, we rigidly demand real-cluster smoke verification before any code is ever tagged for an official release. While the `envtest` framework is exceptional for executing rapid, deterministic integration testing during the standard development loop, it possesses several fatal blind spots: it lacks a true pod scheduler, it does not run a kubelet, and it entirely bypasses realistic, multi-node networking constraints. Therefore, every single release candidate must be manually verified end-to-end on a live, highly available Google Kubernetes Engine cluster. This unforgiving crucible ensures that the controller successfully survives genuine scheduling race conditions, realistic pod image-pull startup delays, and complex identity-aware proxy authentication handshakes that simply cannot be accurately simulated on a developer's laptop. The exact procedure for this rigorous verification is meticulously documented and automated via the smoke-test bash script located within the `hack/` directory of our repository.

## Design Principles

Our engineering decisions are guided by three fundamental, non-negotiable design principles that supersede all other considerations. 

The first principle is relentlessly conservative detection. When our scrapers are attempting to classify highly ambiguous Kubernetes workloads, we strongly and universally prefer to emit false negatives over false positives. If a workload exhibits conflicting or exceptionally ambiguous traits that do not cleanly match our heuristics, the controller will explicitly mark its internal confidence as unresolved and safely decline to classify the pod as an AI workload. We only merge new detection patterns into the codebase when there is a crystal-clear, mathematically definitive signal available in the environment variables or the container arguments. In the high-stakes context of enterprise regulatory compliance, a false positive that hallucinates a nonexistent, highly restricted AI model into an official audit log is significantly more damaging to an auditor's trust than a safely omitted detection.

The second principle is strict, uncompromising cloud neutrality. While the k8s-aibom project was originally incubated within the walls of Google Cloud, it is fundamentally designed as a universally applicable, Kubernetes-native security tool. We absolutely do not permit the introduction of Google-specific dependencies, proprietary vendor APIs, or deeply locked-in SDKs anywhere within the core controller logic. The single, highly scoped exception to this universal rule is the optional Google Cloud Storage external sink, which is completely isolated behind a clean, generic Go interface. The core controller must be able to run flawlessly, without modification, on Amazon Elastic Kubernetes Service, Azure Kubernetes Service, and highly constrained, air-gapped bare-metal on-premises clusters.

The third principle is the per-checkpoint stop-for-review discipline. We require all contributors—including core maintainers—to propose substantial architectural changes, massive refactors, or new scraper designs via a detailed GitHub issue or a brief technical design document before writing significant amounts of code. This highly deliberate friction ensures continuous alignment with our overarching roadmap and prevents enthusiastic contributors from wasting weeks of their valuable time on complex implementations that unknowingly violate our internal security model or our strict backwards-compatibility constraints.

## Pull Request Workflow

Our pull request workflow adheres to the most stringent standard open-source conventions expected of enterprise-grade security tools. To contribute your code to the project, please fork the repository and create a well-named feature branch for your work.  

Before opening a pull request, you must ensure that your code successfully passes the entire integration test suite and that the automated static analysis tools return absolutely zero high-severity or critical security findings. Once your pull request is officially opened against the main branch, our suite of automated GitHub Actions pipelines will rigorously verify your changes against multiple Kubernetes versions. A core maintainer will deeply review your code for architectural soundness, and you must thoroughly address any requested changes before the code can be merged. We require explicit, approving reviews from at least one core maintainer for all merges, without exception.

## Where to File What

If you encounter a reproducible bug, experience a controller crash, or wish to request a completely new feature, please open an Issue on our primary GitHub repository using the appropriate markdown template. If you have a nuanced question about the underlying architecture, want to propose a complex new integration, or simply need assistance debugging your specific cluster configuration, please start a Discussion in the repository's discussions tab where the community can assist you. 

If you discover a vulnerability or a security-critical flaw within the codebase, please do not open a public issue or discussion. Instead, please follow the coordinated disclosure process detailed in our `SECURITY.md` file, which securely routes the report to our maintainery channel for coordination.
